# PRD：医疗健康数据要素行业试跑

## 1. 目标

面向医院、数据集团、保险公司、TPA、保险科技公司、药企和药械企业，连续观察以下变化：

- 国家与地方数据、医疗、医保、药监、网信政策；
- 医疗健康数据流通、授权运营、可信空间、合规与标准；
- 支付方、医院、药企和药械行业中与数据要素相关的变化；
- 数据集团、交易机构、健康数据平台和重点竞品的产品与业务动作；
- 重要行业会议、联盟和标准组织动态。

第一阶段先一次性回填过去 30 天可回源证据，再连续运行 7 个日历日，验证“能否稳定得到相关、中文可读、可聚类、可用于决策的信息”。30 天是历史证据窗口，不是需要空等 30 天的运行期；7 天是自动采集与结果质量的连续验证期。不提前建设知识图谱、复杂趋势指数或自动外部行动。

公开内容目标为约 80% 中国行业信息、20% 国际对标。来源目录数量不等于有效覆盖；只有通过真实采集、内容相关性和原始证据门禁的来源才计入试跑。

## 2. 主要用户

- 业务负责人：判断政策窗口、客户预算方向、合作机会和竞争动作。
- 产品与解决方案负责人：发现医院、保险、药企和药械的真实数据场景。
- 行业研究人员：减少日常收集、去重、回源和周报整理时间。

## 3. 7 天验收

试跑使用 20—30 个高价值公开来源，并输出可审计报告。首次运行必须按明确日期边界回填过去 30 天；随后连续运行 7 天。每条行业主线至少需要 3 个已审核 Event 和两个独立来源才能生成阶段趋势判断；不足时仍展示已采集 Signal、待核验事实和单一权威来源 Event，并明确显示证据缺口。

1. 自动化来源运行成功率达到 90% 或明确列出需替换来源；
2. 能稳定形成绑定多个独立来源的 Event；
3. 人工抽检同一事件聚类准确率可接受；
4. 每条高优先级判断都能回链原始证据；
5. 人工筛选 Top 10 后记录“有决策价值 / 无决策价值”及理由；
6. 记录试跑前后每天信息整理耗时，验证是否节省时间。

增加以下硬门禁：

7. 至少 12 个中文核心来源在 GitHub Actions 中可自动采集，公开内容约 80% 来自中国行业；
8. 有效 Signal 人工抽检相关性准确率达到 85%，无关高优先级 Event 为 0；
9. 聚类精确率达到 90%、召回率达到 80%，至少形成 3 个多来源 Event；
10. 中文站所有公开 Event 拥有中文标题和摘要，英文原题与原始 URL 继续保留；
11. Top 10 至少 7 条被逐条评为具有决策价值；
12. 信息整理时间相对基线至少减少 30%。

页面不得把“尚未形成趋势判断”表达成“没有数据”。证据层必须区分：有效 Signal、待核验事实、单一权威来源 Event、多来源 Event 和趋势判断。Tier 1 原始文件可以形成普通事实 Event 并标记“待交叉验证”；只有高优先级判断和趋势结论才强制要求独立来源互证。

系统自动计算 1、2、4、7、9、10 和候选 Top 10；3、5、6、8、11、12 必须保存逐样本或逐日审计记录，不接受只有聚合百分比、没有样本和理由的结论。

## 4. 非目标

- 不在仓库、快照、日志或公开页面暴露方舟 Coding Plan Key。
- 不把网页可访问等同于允许复制全文；公开快照仍只保存必要元数据、短摘录和原创判断。
- 不激活缺少稳定采集契约的来源；先进入 shadow 并用真实运行证据筛选。
- 不让模型修改来源生命周期、事实分数、发布状态或原始证据。
- 不在第一周宣称趋势计算、知识图谱或生产级行业告警已经完成。
- 不把通用医疗、疾病科普、生活方式、一般公共卫生或纯临床研究自动包装成数据要素 Event。
- 不把单一来源警告当作高优先级 Event 的充分证据。
- 不在行业模式下回退使用上游 AI 行业关键词、实体、评分或 Narrative。

## 5. 完成定义

- Fork 可从上游同步，行业数据不与上游默认快照混用；
- 无模型密钥时可以完成信源审计、采集、导出和 Pages 构建；
- 设置方舟 Coding API Key 后，`glm-5.2` 可以生成通过本地 Schema 与证据门禁的 Event 草稿和周报；
- GitHub Pages 首页只呈现医疗健康数据要素试跑，而不是上游 AI 行业首页；
- 过去 30 天历史证据与连续 7 天验证的公开机器报告、逐条 Top 10 审核和每日耗时记录足以支持继续、调整或停止的决策。
