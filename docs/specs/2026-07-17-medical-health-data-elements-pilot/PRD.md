# PRD：医疗健康数据要素行业试跑

## 1. 目标

面向医院、数据集团、保险公司、TPA、保险科技公司、药企和药械企业，连续观察以下变化：

- 国家与地方数据、医疗、医保、药监、网信政策；
- 医疗健康数据流通、授权运营、可信空间、合规与标准；
- 支付方、医院、药企和药械行业中与数据要素相关的变化；
- 数据集团、交易机构、健康数据平台和重点竞品的产品与业务动作；
- 重要行业会议、联盟和标准组织动态。

第一阶段只验证“能否稳定得到可核验、可聚类、可用于决策的信息”，不提前建设知识图谱、复杂趋势指数或自动外部行动。

## 2. 主要用户

- 业务负责人：判断政策窗口、客户预算方向、合作机会和竞争动作。
- 产品与解决方案负责人：发现医院、保险、药企和药械的真实数据场景。
- 行业研究人员：减少日常收集、去重、回源和周报整理时间。

## 3. 7 天验收

试跑使用 20—30 个高价值公开来源，并输出可审计报告：

1. 自动化来源运行成功率达到 90% 或明确列出需替换来源；
2. 能稳定形成绑定多个独立来源的 Event；
3. 人工抽检同一事件聚类准确率可接受；
4. 每条高优先级判断都能回链原始证据；
5. 人工筛选 Top 10 后记录“有决策价值 / 无决策价值”及理由；
6. 记录试跑前后每天信息整理耗时，验证是否节省时间。

系统自动计算 1、2、4 和候选 Top 10；3、5、6 保留明确的人工验收字段，不伪造自动结论。

## 4. 非目标

- 不在仓库、快照、日志或公开页面暴露方舟 Coding Plan Key。
- 不把网页可访问等同于允许复制全文；公开快照仍只保存必要元数据、短摘录和原创判断。
- 不激活缺少稳定采集契约的来源；先进入 shadow 并用真实运行证据筛选。
- 不让模型修改来源生命周期、事实分数、发布状态或原始证据。
- 不在第一周宣称趋势计算、知识图谱或生产级行业告警已经完成。

## 5. 完成定义

- Fork 可从上游同步，行业数据不与上游默认快照混用；
- 无模型密钥时可以完成信源审计、采集、导出和 Pages 构建；
- 设置方舟 Coding API Key 后，`glm-5.2` 可以生成通过本地 Schema 与证据门禁的 Event 草稿和周报；
- GitHub Pages 首页只呈现医疗健康数据要素试跑，而不是上游 AI 行业首页；
- 连续 7 天的公开机器报告足以支持继续、调整或停止的决策。
