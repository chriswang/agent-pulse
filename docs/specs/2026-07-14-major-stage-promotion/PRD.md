# PRD：重大事件驱动的趋势阶段晋级

## 1. 问题

当前阶段轨迹是代码内的人工叙事。新的公开 Event 和 Evidence 会进入最后一个开放阶段，但即使出现真正改变行业判断的热点、产品转折或 AGI 里程碑，系统也不会形成新阶段。反过来，如果简单地让每个高分 Event 自动新增阶段，又会把阶段退化成新闻标签，破坏趋势判断的稀缺性和可信度。

用户重点关注的不是“又有一条新闻”，而是：为什么旧阶段已经不足以解释现实、什么事实触发了阶段变化、影响谁、有哪些反证、下一步观察什么。

## 2. 目标

- 极低频识别足以改变六条趋势任一阶段判断的重大事件或 AGI 里程碑。
- 先用确定性规则筛选，再由 DeepSeek V4 Pro 基于公开 Evidence 做结构化阶段判断；模型不能单独决定资格。
- 新阶段必须绑定已公开 Event、原始 Evidence、独立 Source、明确时间边界和可证伪下一信号。
- 新阶段出现时，创建或幂等更新一个专属 GitHub Issue，重点说明事件、影响、阶段变化理由、证据和风险。
- 阶段持久化进入受版本控制的公开安全 JSON，并随 Data Refresh 回流仓库、触发 Pages。
- AI、Issue 或阶段写入失败不能丢失已经完成的确定性增量采集。

## 3. 非目标

- 不按天、周或 Event 数量机械切阶段。
- 不把聚合站热度、人工分数或单一模型判断当作阶段事实。
- 不允许模型修改历史阶段、删除阶段或重写 Event/Evidence。
- 不在输入中使用 review Event、原始 payload、私有备注或 Source Catalog 观察项冒充事实证据。
- 不在一次运行中创建多个阶段。
- 不因模型 `hold` 或结构失败创建“伪阶段” Issue。

## 4. 用户体验

当新阶段真实成立时：

1. 趋势轨迹追加一个有明确起始日期的新阶段；上一开放阶段在新阶段前一天结束。
2. 新阶段展示阶段摘要、转折解释、下一验证信号；原 Event 仍是唯一事实节点。
3. 与触发事件关联的 Evidence 和 Source 通过既有阶段证据模块进入新阶段，不复制事实。
4. GitHub Issue 使用 `stage:milestone` 标签，包含：
   - 哪个趋势发生变化；
   - 发生了什么；
   - 为什么旧阶段不再足够；
   - 对 CEO、投资、技术与产品决策的主要影响；
   - 原始 Evidence / Source 表；
   - 反向信号与下一验证；
   - 模型、置信度、输入哈希和 Actions 运行链接。

没有阶段级变化时，页面、阶段更新时间和 Issue 数量都保持不变。

## 5. 验收标准

- 确定性门禁不通过时不调用 DeepSeek。
- 只有 `deepseek-v4-pro`、thinking enabled、high reasoning effort 用于阶段任务；Event 与周报原模型不被隐式替换。
- V4 Pro 返回 `hold` 时不写文件、不建 Issue。
- V4 Pro 返回 `promote` 仍须通过本地 Schema、URL/slug allowlist、来源独立性、时间边界、重复和冷却期门禁。
- 每个晋级至少包含一个 Tier 1 原始来源、两个不同来源与两个不同主机的公开 Evidence。
- 每条趋势 45 天内最多新增一个阶段；每次 Data Refresh 全局最多新增一个阶段。
- Issue 创建/更新成功后才允许把该候选写入阶段文件；记录必须回链 Issue。
- 阶段文件、快照和公开 DTO 通过隐私扫描；原 prompt、reasoning 和原始 completion 永不提交或上传。
- 新阶段改变 public fingerprint、提交回 `main` 并触发 Pages。
- 任一可选阶段步骤失败只留下 warning 与安全 artifact，不阻断 snapshot 回流。
